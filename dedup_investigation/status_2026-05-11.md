## Open issue — rate_equation regression suspected

While investigating same-hash-different-loss cases at n=7 (loss
range 0.0178–0.0194 within one canonical hash bucket), found:

1. **Canonicalizer is correct.** Both mechs render to byte-identical
   v lines. Only the user-defined vs Wegscheider sections differ
   in WHICH symbol got aliased to which representative. Same
   underlying equation.
2. **`fitted_params` bug fixed in `c8a3302`.** Before the fix,
   Pass-2-absorbed symbols (K9 → K4, K6 → K3, etc.) leaked into
   the fittable parameter set. Different mechs in the same hash
   bucket exposed different dummy dimensions to CMA-ES, causing
   noisy convergence. After the fix, both mechs have identical
   `fitted_params = (K1,K2,K3,K4,k10f,L)` — same 6-dim search
   space.
3. **BUT the saved CSV losses still show 8.6% spread.** Those
   losses were recorded under the old code (with leaked dummy
   dims). The 8.6% gap *might* just be the dummy-dim bug now
   fully addressed by c8a3302. Or it might be **partly** that AND
   **partly** the always-expanded rate-equation regression
   (Commit 2): per-call eval went from ~600ns (factored) to
   ~2µs (always-expanded), making `maxtime=60.0s` more
   marginal under CMA-ES.

### If rate_equation regression is confirmed

The always-expanded polynomial form intrinsically does more
arithmetic per call than the factored form did. Possible
mitigations:

- **Reintroduce factoring**, scoped to runtime-only (don't print
  factored; just emit a factored Expr for `@generated
  rate_equation`). Keep canonicalizer string output expanded so
  Source-A dedup still works.
- **Common-subexpression elimination** in the `_poly_to_expr`
  output: detect repeated `(1 + S/K1)` patterns and assign them
  to local lets in the body.
- **Hand-optimized arithmetic ordering**: factor out
  `1/(p_2 * p_3)` etc. to avoid recomputation.

The performance test was relaxed from `allocs == 0` /
`t < 100 ns` to `allocs < 4 KiB` / `t < 10 µs` in Commit 2. If
maxtime=60s is genuinely insufficient under the new arithmetic
load, that relaxation has user-visible consequences and should
be revisited.

## Update — issue (1) closed

Root cause of the rate_equation alloc/slowdown regression: `_nest_binary` in `src/sym_poly_for_rate_eq_derivation.jl` was emitting flat n-ary `+(a, b, …, a_N)` Expr nodes. Above ~30 terms Julia's compiler stops inlining the varargs `+` path and falls back to a heap-boxed-argument-tuple codegen — turning the per-call cost from 100 ns / 0 B into ~1 µs / ~2 KB. Same regression as commit `598a964` (Feb), reverted in `dec0707` (Mar), re-exposed by commit `28041fe` on this branch when factoring was dropped. Fixed by restoring the balanced-binary emission. See `docs/superpowers/specs/2026-05-11-rate-eq-emission-perf-fix-design.md` and `docs/superpowers/plans/2026-05-11-rate-eq-emission-perf-fix.md`.

Issue (2) — residual same-hash-different-loss / same-loss-different-hash within `params_estimate_*.csv` — is deferred to a follow-up. The hypothesis in this status doc ("CMA-ES `maxtime=60` marginal under the slower per-call cost") is now testable: re-fit a same-hash mech pair against `LDH_data.csv` with the perf fix applied, at the default `maxtime`. If the within-hash spread collapses to fitter noise, the c8a3302 `fitted_params` filter fix is the only remaining substantive change and the dedup work is done. If anything residual remains, dump canonical strings for two non-collapsing mechanisms and diff to identify the next quirk.
